ADR · 0001

PR 생성 직후
worktree 제거

git은 한 브랜치를 하나의 worktree에만 체크아웃할 수 있다. 리뷰 단계에서 main checkout이 feature branch에 진입하지 못하는 문제를 해소하기 위해, PR 생성 직후 worktree를 즉시 제거한다.

Status · Accepted 2026-05-16 tag · worktree tag · submodule tag · superpowers-workflow

01Context

chatbot 레포는 backend / frontend submodule 작업 시 .worktrees/<branch> 형태의 git worktree를 만들어 격리된 환경에서 작업한다. 표준 흐름은 worktree에서 commit → push → gh pr create이다.

git의 근본 제약은 다음과 같다:

한 브랜치는 동시에 하나의 worktree에서만 checkout될 수 있다.

따라서 worktree가 feat/foo를 점유하는 동안 main checkout에서 git switch feat/foo를 시도하면 다음 오류로 차단된다: 

fatal: 'feat/foo' is already checked out at '/.../.worktrees/feat/foo'

리뷰 피드백 대응이나 추가 디버깅을 main checkout(IDE·dev 서버가 묶여 있는 1차 작업공간)에서 진행하고 싶을 때 이 잠금이 작업을 직접적으로 막는다.

02Decision

gh pr create가 성공한 직후 다음을 실행한다:

Run git worktree remove .worktrees/<branch>

이후 동일 브랜치에 대한 모든 추가 작업(리뷰 피드백, fix-up, follow-up commit)은 main checkout에서 git switch <branch> 후 직접 진행한다. worktree를 다시 만들지 않는다.

이 룰은 backend / frontend submodule 양쪽에 동일하게 적용된다.

03Before / After scenarios

변경 전과 후의 동작을 단계별 시나리오로 비교한다.

Before worktree 유지 정책

PR 생성 후에도 worktree가 feat/foo를 점유

1. cd backend
2. git worktree add .worktrees/feat/foo \
     -b feat/foo origin/main
3. (worktree) 작업 + commit + push
4. (worktree) gh pr create
5. PR 리뷰어가 수정 요청 댓글 작성
6. cd /path/to/backend          # main checkout
7. git switch feat/foo
   → fatal: 'feat/foo' is already
     checked out at .worktrees/...   ✗
8. (어쩔 수 없이) cd .worktrees/feat/foo
   → 수정 → commit → push
9. main checkout은 PR이 merge될 때까지
   feature branch에 진입 불가

문제 지점

  • main checkout이 1차 작업공간임에도 feature branch 진입 불가
  • worktree 경로를 기억해 매번 cd 해야 함
  • .worktrees/ 디렉토리가 점점 무거워짐

After PR 생성 직후 제거

PR이 만들어진 순간 잠금 해제

1. cd backend
2. git worktree add .worktrees/feat/foo \
     -b feat/foo origin/main
3. (worktree) 작업 + commit + push
4. (worktree) gh pr create
5. cd /path/to/backend          # main checkout
6. git worktree remove \
     .worktrees/feat/foo             ✓ NEW
7. PR 리뷰어가 수정 요청 댓글 작성
8. git switch feat/foo               ✓ 정상 진입
9. 수정 → commit → push
   (main checkout에서 직접)
10. PR merge 후
    git switch main && git branch -d feat/foo

개선 지점

  • main checkout에서 feature branch에 자유롭게 진입
  • .worktrees/ 누적 없음 (PR마다 생성·제거 1회)
  • 리뷰 피드백 사이클이 main checkout 안에서 닫힘

04Consequences

이 결정의 트레이드오프.

Positive

  • main checkout에서 모든 git 동작 가능 — 브랜치 잠금 충돌 제거
  • 멀티 worktree 누적으로 인한 디스크·인덱스 비용 감소
  • 리뷰 피드백 → 추가 commit 흐름 단순화 (cd 불필요)
  • IDE·dev 서버가 항상 main checkout을 가리키는 일관성 유지

Negative

  • worktree의 "다른 브랜치 작업 중에도 격리 유지" 효과가 PR 생성 시점까지로 한정된다.
  • 리뷰 단계 이후 main checkout이 feature branch에 묶이므로, 동시에 다른 작업을 하려면 stash 또는 별도 worktree 재생성이 필요하다.
  • 룰을 잊고 worktree를 남기면 잠금 문제가 재발 — CLAUDE.md / .claude/rules/로 보강 필요.

05Alternatives considered

다음 대안들을 검토했으나 모두 본 문제를 해결하지 못해 기각했다.

대안 기각 사유
worktree 유지, 리뷰 피드백도 worktree에서 처리 원 문제(main checkout 잠금)가 그대로 남음. 사용자가 명시적으로 거부.
worktree 미사용 — main checkout에서 feature branch 직접 작업 작업 중 main으로 잠깐 돌아가야 할 때 stash 필요 → 격리 이점 상실. 병렬 implementer subagent 흐름과도 충돌.
worktree 제거 시점을 PR merge 이후로 늦춤 리뷰 사이클(보통 수 시간~수일) 동안 잠금 문제 지속 → 본 문제 미해결.
같은 ref를 두 worktree에 동시 노출 git이 해당 기능을 제공하지 않음.
