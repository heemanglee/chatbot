Implementation Spec · Chat Date Context

클라이언트 시간대 기준으로 오늘 날짜를 서버에 전달한다

사용자가 KST 오전에 “내일”을 물었을 때 서버 UTC 날짜로 해석되는 문제를 막기 위한 구현 계약이다. 요청 단위의 사용자 로컬 날짜 컨텍스트를 프롬프트와 검색 쿼리 정규화에 동일하게 적용한다.

Backend touchpointbackend/app/domain/chat/service.py
Contract fieldclient_timezone + client_today

§ 1문제 정의

현재 서버는 채팅 turn의 시스템 프롬프트를 만들 때 UTC 기준 날짜를 주입한다. 이 때문에 사용자의 실제 로컬 날짜와 UTC 날짜가 다른 시간대 경계에서 “오늘”, “내일”, “어제”가 잘못 풀린다.

Current behavior

서버 UTC 기준 날짜

KST 2026-06-07 07:47은 UTC 2026-06-06 22:47이다. 프롬프트가 오늘을 6월 6일로 선언하면 “내일”은 6월 7일로 해석된다.

Target behavior

사용자 로컬 기준 날짜

클라이언트 시간대가 Asia/Seoul이면 오늘은 2026-06-07이고 “내일”은 2026-06-08이다.

§ 3API 계약

클라이언트는 브라우저의 IANA timezone과 그 timezone에서 계산한 오늘 날짜를 함께 보낸다. 서버는 timezone을 신뢰 가능한 계산 기준으로 사용하고, client_today는 관측 가능한 보조 입력과 테스트 편의 값으로 검증한다. 

{
  "message": "내일부터 일주일간 판교 날씨 표로 정리",
  "tools": { "web_search": true },
  "client_timezone": "Asia/Seoul",
  "client_today": "2026-06-07"
}

client_timezone은 필수에 가깝게 다루되, 레거시 클라이언트 호환을 위해 optional로 시작한다. 값이 없거나 유효하지 않으면 서버는 UTC로 fallback하고, 로그에 fallback 사유를 남긴다.

필드타입검증사용처
client_timezone string | null zoneinfo.ZoneInfo로 검증 가능한 IANA ID 현재 local datetime 계산, 프롬프트 날짜 라인, 검색어 정규화
client_today YYYY-MM-DD | null ISO date 형식. 서버 계산값과 다르면 서버 계산값 우선 디버깅, 테스트, 클라이언트 관측값 기록

§ 4실행 흐름

날짜 기준은 요청 생성 시점부터 답변 생성 직전까지 같은 값으로 유지되어야 한다. 한 turn 안에서 중간에 날짜가 바뀌면 검색과 답변이 다른 기준일을 사용할 수 있기 때문이다.

§ 5구현 지침

Frontend

요청 body enrich

function getClientDateContext() {
  const timezone =
    Intl.DateTimeFormat().resolvedOptions().timeZone || "UTC";
  const today = new Intl.DateTimeFormat("en-CA", {
    timeZone: timezone,
    year: "numeric",
    month: "2-digit",
    day: "2-digit",
  }).format(new Date());
  return { client_timezone: timezone, client_today: today };
}

body: JSON.stringify({
  ...payload,
  ...getClientDateContext(),
})
Backend

Turn date context

from zoneinfo import ZoneInfo, ZoneInfoNotFoundError

def resolve_client_now(client_timezone: str | None) -> datetime:
    try:
        tz = ZoneInfo(client_timezone or "UTC")
    except ZoneInfoNotFoundError:
        tz = timezone.utc
    return datetime.now(tz)

build_system_prompt()의 인자는 today보다 의미가 넓은 current_datetime 또는 date_context로 바꾸는 편이 좋다. 날짜뿐 아니라 시간과 timezone label까지 프롬프트에 포함해야 새벽 경계 문제를 설명할 수 있다.

§ 6검색 쿼리 정규화

답변 프롬프트만 고치면 모델의 최종 답변은 개선될 수 있지만, 검색 서비스에 전달되는 질의가 여전히 내일이면 외부 검색기가 다른 날짜 기준으로 해석할 수 있다. 따라서 Agentic RAG의 contextualize 단계에도 같은 날짜 컨텍스트를 넣는다.

입력기준 컨텍스트검색 질의 목표
내일부터 일주일간 판교 날씨 Asia/Seoul, 2026-06-07 2026년 6월 8일부터 2026년 6월 14일까지 판교 날씨
어제 OpenAI 뉴스 America/Los_Angeles, 2026-06-06 2026년 6월 5일 OpenAI 뉴스 
CONTEXTUALIZE_QUERY_PROMPT 추가 규칙:
- Interpret relative date words using the provided user local date context.
- Rewrite relative dates such as 오늘, 내일, 어제, 이번주 into absolute dates.
- Preserve the user's locale and language in the final search_query.

현재 단일 질문은 contextualize를 생략한다. 최신 정보 질문이면서 상대 날짜 표현이 있으면, 단일 질문이어도 contextualize를 수행하도록 조건을 열어야 한다.

§ 7검증 기준

테스트조건기대 결과
BE unit now_utc=2026-06-06T22:47:00Z, client_timezone=Asia/Seoul 프롬프트가 2026-06-07과 Asia/Seoul을 포함한다.
BE unit 잘못된 timezone 문자열 UTC fallback, 예외 없이 SSE stream 유지, warning log 기록.
Agentic RAG unit 내일 판교 날씨 + Asia/Seoul local today 2026-06-07 검색 시작 progress query가 2026년 6월 8일을 포함한다.
FE unit streamChat, streamPrivateChat 호출 body에 client_timezone, client_today가 포함된다.
Rollout note

레거시 클라이언트 호환
